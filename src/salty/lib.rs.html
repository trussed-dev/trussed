1#![no_std]
2/*!
3Mashup of [TweetNaCl](https://tweetnacl.cr.yp.to/) with
4[ed25519-dalek](https://lib.rs/crates/ed25519-dalek)
5aimed towards embedded use cases on microcontrollers.
6
7For more context, see also the [salty book](https://book.salty.rs).
8
9Originally, this library was a transliteration of the C implementation of
10Ed25519 signatures in TweetNaCl to Rust, "with helpful explanations".
11
12Iterating over the not-very-nice API surface of NaCl, we ended up with
13a close relative of the "[dalek](https://dalek.rs)" APIs, where things are modeled as,
14for instance, "compressed y-coordinate of an Edwards25519 curve point",
15instead of raw bytes.
16
17One reason the current ed25519-dalek library in its current state is not
18ideal for microcontrollers is that it includes ~40kB of pre-computed data
19to speed things up. Moreover, its implementations are optimized for PC.
20
21## Usage
22
23The main entry point of the API is either a keypair, or a public key.
24
25For keypairs, an external trusted source of entropy is assumed, letting
26us deterministically construct a keypair as:
27
28```
29let seed: [u8; 32] = [42; 32]; // 32 actually entropic bytes
30let keypair: salty::Keypair = salty::Keypair::from(&seed);
31```
32
33Any byte slice of data that fits in memory can then be signed
34deterministically via
35
36```
37# let seed: [u8; 32] = [42; 32]; // 32 actually entropic bytes
38# let keypair: salty::Keypair = salty::Keypair::from(&seed);
39let data: &[u8] = &[1, 2, 3]; // some data
40let signature: salty::Signature = keypair.sign(data);
41```
42
43Thereafter, the signature can be checked:
44
45```
46# let seed: [u8; 32] = [42; 32]; // 32 actually entropic bytes
47# let keypair: salty::Keypair = salty::Keypair::from(&seed);
48# let data: &[u8] = &[1, 2, 3]; // some data
49# let signature: salty::Signature = keypair.sign(data);
50let public_key: salty::PublicKey = keypair.public;
51assert!(public_key.verify(data, &signature).is_ok());
52```
53
54For serialization purposes, the entropic seed *is* the private key (32 bytes).
55Both public keys and signatures have `to_bytes()` methods, returning 32 and 64
56bytes, respectively.
57
58```
59# let seed: [u8; 32] = [42; 32]; // 32 actually entropic bytes
60# let keypair: salty::Keypair = salty::Keypair::from(&seed);
61# let data: &[u8] = &[1, 2, 3]; // some data
62# let signature: salty::Signature = keypair.sign(data);
63# let public_key = &keypair.public;
64let serialized_public_key: [u8; 32] = public_key.to_bytes();
65let serialized_signature: [u8; 64] = signature.to_bytes();
66```
67
68Conversely, `PublicKey` implements `TryFrom` (verifying the alleged point actually
69lies on the curve), and `Signature` implements `From`.
70
71```
72# let seed: [u8; 32] = [42; 32]; // 32 actually entropic bytes
73# let keypair: salty::Keypair = salty::Keypair::from(&seed);
74# let data: &[u8] = &[1, 2, 3]; // some data
75# let signature: salty::Signature = keypair.sign(data);
76# let public_key = &keypair.public;
77# let serialized_public_key: [u8; 32] = public_key.to_bytes();
78# let serialized_signature: [u8; 64] = signature.to_bytes();
79use core::convert::TryInto;
80let deserialized_public_key: salty::PublicKey = (&serialized_public_key).try_into().unwrap();
81let deserialized_signature: salty::Signature = (&serialized_signature).into();
82assert!(deserialized_public_key.verify(data, &deserialized_signature).is_ok());
83```
84
85Please note that `Ed25519` signatures are *not* init-update-finalize signatures,
86since two passes over the data are made, sequentially (the output of the first pass
87is an input to the second pass).
88For cases where the data to be signed does not fit in memory, as explained in
89[RFC 8032](https://tools.ietf.org/html/rfc8032/) an alternative algorithm `Ed25519ph` ("ph" for prehashed) is
90defined. This is *not* the same as applying Ed25519 signature to the SHA512 hash of
91the data; it is is exposed via `Keypair::sign_prehashed` and
92`PublicKey::verify_prehashed`. Additionally, there is the option of using "contexts"
93for both regular and prehashed signatures.
94
95## Features
96The bulk of time generating and verifying signatures is spent with field operations
97in the base field of the underlying elliptic curve. This library has two implementations:
98The `tweetnacl` implementation is portable but quite slow, and the `haase` implementation,
99which makes use of the `UMAAL` assembly instruction, which is only available on
100Cortex-M4 and Cortex-M33 microcontrollers. By default, on these targets the fast implementation
101is selected, the `tweetnacl` variant can be triggered with the `slow-motion` feature.
102
103This `UMAAL` operation is a mapping `(a, b, c, d) ⟼ a*b + c + d`, where the inputs are `u32`
104and the output is a `u64` (there is no overflow). In the future, we hope to offer a third
105implementation, which would do "schoolbook multiplication", but using this operation, e.g.
106as a compiler intrinsic. The idea is to have a similarly speedy implementation without the
107obscurity of the generated assembly code of the `haase` implementation.
108
109Current numbers on an NXP LPC55S69 running at 96Mhz, with "tweetnacl" implementation:
110- signing prehashed message: 52,632,954 cycles
111- verifying said message: 100,102,158 cycles
112- code size for this: 19,724 bytes
113
114Obviously, this needed to improve.
115
116Current numbers on an NXP LPC55S69 running at 96Mhz, with "haase" implementation:
117- signing prehashed message: 8,547,161 cycles
118- verifying said message: 16,046,465 cycles
119- code size: similar
120
121In both cases, we suggest compiling with at least minimal optimization, to get rid
122of the zero-cost abstractions.
123
124## Future
125
126Future plans include:
127- rigorous correctness checks
128- rigorous checks against timing side-channels, using the DWT cycle count of ARM MCUs
129- ensure dropped secrets are `zeroize`d
130- add the authenticated encryption part of NaCl
131- add X25519, i.e., Diffie-Hellman key agreement
132- speedy yet understandable field operations using `UMAAL`
133
134*/

141/// Extensible error type for all `salty` operations.
142///
143/// This enum has a member `NoError` with value zero, for use in the C API.
144#[derive(Clone, Copy, Debug, Eq, Hash, PartialEq)]
145#[repr(C)]
146#[non_exhaustive]
147pub enum Error {
148    /// Never occurs, simplifies C bindings
149    NoError = 0,
150
151    /// Bytes do not correspond to a canonical base field element
152    NonCanonicalFieldElement,
153
154    /// Public key bytes invalid
155    PublicKeyBytesInvalid,
156
157    /// Signature verification failed
158    SignatureInvalid,
159
160    /// Context for prehashed signatures too long
161    ContextTooLong,
162
163    /// Point is on other twist of curve
164    WrongTwist,
165}
166
167/// Result type for all `salty` operations.
168pub type Result<T = ()> = core::result::Result<T, Error>;
