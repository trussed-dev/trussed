1//! # Postcard
2//!
3//! Postcard is a `#![no_std]` focused serializer and deserializer for Serde.
4//!
5//! Postcard aims to be convenient for developers in constrained environments, while
6//! allowing for flexibility to customize behavior as needed.
7//!
8//! ## Design Goals
9//!
10//! 1. Design primarily for `#![no_std]` usage, in embedded or other constrained contexts
11//! 2. Support a maximal set of `serde` features, so `postcard` can be used as a drop in replacement
12//! 3. Avoid special differences in code between communication code written for a microcontroller or a desktop/server PC
13//! 4. Be resource efficient - memory usage, code size, developer time, and CPU time; in that order
14//! 5. Allow library users to customize the serialization and deserialization  behavior to fit their bespoke needs
15//!
16//! ## Variable Length Data
17//!
18//! Variable length data (such as slices) are prefixed by their length.
19//!
20//! Length is encoded as a [Varint]. This is done for two reasons: to minimize wasted bytes
21//! on the wire when sending slices with items less than 127 items (typical for embedded),
22//! and to reduce compatibility issues between 32-bit and 64-bit targets due to differing sizes
23//! of `usize`.
24//!
25//! Similarly, `enum` descriminants are encoded as varints, meaning that any enum with less than
26//! 127 variants will encode its discriminant as a single byte (rather than a `u32`).
27//!
28//! Varints in `postcard` have a maximum value of the usize for that platform. In practice, this
29//! means that 64-bit targets should not send messages with slices containing `(1 << 32) - 1` items
30//! to 32-bit targets, which is uncommon in practice. Enum discriminants already have a fixed
31//! maximum value of `(1 << 32) - 1` as currently defined in Rust. Varints larger than the current platform's
32//! `usize` will cause the deserialization process to return an `Err`.
33//!
34//! [Varint]: https://developers.google.com/protocol-buffers/docs/encoding
35//!
36//! ## Example - Serialization/Deserialization
37//!
38//! Postcard can serialize and deserialize messages similar to other `serde` formats.
39//!
40//! Using the default `heapless` feature to serialize to a `heapless::Vec<u8>`:
41//!
42//! ```rust
43//! # #[cfg(feature = "heapless")] {
44//! use core::ops::Deref;
45//! use serde::{Serialize, Deserialize};
46//! use postcard::{from_bytes, to_vec};
47//! use heapless::Vec;
48//!
49//! #[derive(Serialize, Deserialize, Debug, Eq, PartialEq)]
50//! struct RefStruct<'a> {
51//!     bytes: &'a [u8],
52//!     str_s: &'a str,
53//! }
54//! let message = "hElLo";
55//! let bytes = [0x01, 0x10, 0x02, 0x20];
56//! let output: Vec<u8, 11> = to_vec(&RefStruct {
57//!     bytes: &bytes,
58//!     str_s: message,
59//! }).unwrap();
60//!
61//! assert_eq!(
62//!     &[0x04, 0x01, 0x10, 0x02, 0x20, 0x05, b'h', b'E', b'l', b'L', b'o',],
63//!     output.deref()
64//! );
65//!
66//! let out: RefStruct = from_bytes(output.deref()).unwrap();
67//! assert_eq!(
68//!     out,
69//!     RefStruct {
70//!         bytes: &bytes,
71//!         str_s: message,
72//!     }
73//! );
74//! # }
75//! ```
76//!
77//! Or the optional `alloc` feature to serialize to an `alloc::vec::Vec<u8>`:
78//! ```rust
79//! # #[cfg(feature = "alloc")] {
80//! use core::ops::Deref;
81//! use serde::{Serialize, Deserialize};
82//! use postcard::{from_bytes, to_allocvec};
83//! extern crate alloc;
84//! use alloc::vec::Vec;
85//!
86//! #[derive(Serialize, Deserialize, Debug, Eq, PartialEq)]
87//! struct RefStruct<'a> {
88//!     bytes: &'a [u8],
89//!     str_s: &'a str,
90//! }
91//! let message = "hElLo";
92//! let bytes = [0x01, 0x10, 0x02, 0x20];
93//! let output: Vec<u8> = to_allocvec(&RefStruct {
94//!     bytes: &bytes,
95//!     str_s: message,
96//! }).unwrap();
97//!
98//! assert_eq!(
99//!     &[0x04, 0x01, 0x10, 0x02, 0x20, 0x05, b'h', b'E', b'l', b'L', b'o',],
100//!     output.deref()
101//! );
102//!
103//! let out: RefStruct = from_bytes(output.deref()).unwrap();
104//! assert_eq!(
105//!     out,
106//!     RefStruct {
107//!         bytes: &bytes,
108//!         str_s: message,
109//!     }
110//! );
111//! # }
112//! ```
113//!
114//! ## Example - Flavors
115//!
116//! `postcard` supports a system called `Flavors`, which are used to modify the way
117//! postcard serializes or processes serialized data. These flavors act as "plugins" or "middlewares"
118//! during the serialization process, and can be combined to obtain complex protocol formats.
119//!
120//! Here, we serialize the given data, while simultaneously encoding it using COBS (a "modification flavor"),
121//! and placing the output in a byte slice (a "storage flavor").
122//!
123//! Users of `postcard` can define their own Flavors that can be combined with existing Flavors.
124//!
125//! ```rust
126//! use postcard::{
127//!     serialize_with_flavor,
128//!     flavors::{Cobs, Slice},
129//! };
130//!
131//! let mut buf = [0u8; 32];
132//!
133//! let data: &[u8] = &[0x01, 0x00, 0x20, 0x30];
134//! let buffer = &mut [0u8; 32];
135//! let res = serialize_with_flavor::<[u8], Cobs<Slice>, &mut [u8]>(
136//!     data,
137//!     Cobs::try_new(Slice::new(buffer)).unwrap(),
138//! ).unwrap();
139//!
140//! assert_eq!(res, &[0x03, 0x04, 0x01, 0x03, 0x20, 0x30, 0x00]);
141//! ```
142//!
143//! ## Setup - `Cargo.toml`
144//!
145//! Don't forget to add [the `no-std` subset](https://serde.rs/no-std.html) of `serde` along with `postcard` to the `[dependencies]` section of your `Cargo.toml`!
146//!
147//! ``` toml
148//! # Cargo.toml
149//!
150//! [dependencies]
151//! postcard = "0.7.2"
152//!
153//! # By default, `serde` has the `std` feature enabled, which makes it unsuitable for embedded targets
154//! # disabling default-features fixes this
155//! serde = { version = "1.0.*", default-features = false }
156//! ```
