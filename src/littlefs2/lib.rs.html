6/*!
7
8[littlefs](https://github.com/littlefs-project/littlefs) is a filesystem for microcontrollers
9written in C, that claims to be *fail-safe*:
10- power-loss resilience, by virtue of copy-on-write guarantees
11- bounded RAM/ROM, with stack-allocated buffers
12
13Since [version 2](https://github.com/littlefs-project/littlefs/releases/tag/v2.0.0), it has
14some nifty features such as:
15- dynamic wear-leveling, including detection of bad Flash blocks
16- custom user attributes
17- inline files, avoiding block waste
18
19For more background, see its [design notes](https://github.com/littlefs-project/littlefs/blob/master/DESIGN.md)
20and the [specification](https://github.com/littlefs-project/littlefs/blob/master/SPEC.md) of its format.
21
22### What is this?
23
24This library, [`littlefs2`](https://lib.rs/littlefs2), offers an idiomatic Rust API for littlefs.
25
26It follows the design of [`std::fs`](https://doc.rust-lang.org/std/fs/index.html) as much as reasonable,
27and builds on the bindings [`littlefs2-sys`](https://lib.rs/littlefs2-sys).
28
29Some complications arise due to the lack of const generics in Rust, we work around these
30with the [`generic-array`](https://lib.rs/generic-array) library, and long for the day when
31constants associated to traits will be treated as constants by the compiler.
32
33Another complication is the fact that files (and directories) need to be closed before they go out of scope,
34since the main littlefs state structure contains a linked list which would exhibit UB (undefined behaviour)
35otherwise, see [issue #3](https://github.com/trussed-dev/littlefs2/issues/3) and
36[issue #5](https://github.com/trussed-dev/littlefs2/issues/5). We choose *not* to call `close` in `drop` (as
37`std::fs` does), since these operations could panic if for instance `littlefs` detects Flash corruption
38(from which the application might otherwise recover).
39
40For this reason, the various `File`-related `open` methods are marked as `unsafe`.
41Instead, a closure-based API is offered (`open_and_then` and friends),
42the same is done for `Filesystem::read_dir`. Under the hood, this API first calls the unsafe constructor,
43then calls the user-supplied closure, and finally closes the object.
44

49### Usage
50
51To use this library, implement `littlefs2::driver::Storage`.
52The macro `ram_storage!` generates examples of this.
53
54Roughly speaking, the [`Storage`](driver/trait.Storage.html) trait defines a block device in
55terms of actual and `typenum` constants, and an implementation supplies methods to read, erase and write.
56
57The filesystem and each open file need memory for state and caching, this has to be allocated
58beforehand and passed to constructors.
59
60### `no_std`
61
62This library is `no_std` compatible, but there are two gotchas.
63
64- The dev-dependency `memchr` of `littlefs2-sys` has its `std` features activated. To prevent this, upgrade to at least Rust 1.51
65  and add `resolver = "2"` in the consuming code's `[package]` section. This will be the default in Rust 2021 edition.
66
67- At link time, `lfs.c` has a dependency on `strcpy`. When not linking to a `libc` with this symbol, activate the `c-stubs` feature
68  to provide an implementation.
69
70### Design notes
71
72All operations on the filesystem require passing a `&mut Storage`, which guarantees by Rust's
73borrow checker that only one thread can manipulate the filesystem.
74This design choice (as opposed to consuming the Storage, which would be less verbose) was made to
75enable use of the underlying flash peripheral outside of the filesystem (the `Storage` can be
76dropped and reconstructed).  For instance, one could setup an additional filesystem,
77or handle some flash data manually.
78
79As an experiment, we implemented [`ReadDirWith`](fs/struct.ReadDirWith.html). It converts a
80[`ReadDir`](fs/struct.ReadDir.html) (which needs mutable references, and so is "not quite an iterator"
81over the files of a directory), into a true iterator, by temporarily binding the mutable references.
82
83Currying with lifetime gymnastics!
84
85In the future, we may extend this approach to other operations, thus adding a secondary API layer.
86
87<https://play.rust-lang.org/?edition=2018&gist=c86abf99fc87551cfe3136e398a45d19>
88
89Separately, keeping track of the allocations is a chore, we hope that
90[`Pin`](https://doc.rust-lang.org/core/pin/index.html) magic will help fix this.
91
92### Example
93
94```
95# use littlefs2::fs::{Filesystem, File, OpenOptions};
96# use littlefs2::io::prelude::*;
97# use littlefs2::path;
98#
99# use littlefs2::{consts, ram_storage, driver, io::Result};
100#
101#
102// example storage backend
103ram_storage!(tiny);
104let mut ram = Ram::default();
105let mut storage = RamStorage::new(&mut ram);
106
107// must format before first mount
108Filesystem::format(&mut storage).unwrap();
109// must allocate state statically before use
110let mut alloc = Filesystem::allocate();
111let mut fs = Filesystem::mount(&mut alloc, &mut storage).unwrap();
112
113// may use common `OpenOptions`
114let mut buf = [0u8; 11];
115fs.open_file_with_options_and_then(
116    |options| options.read(true).write(true).create(true),
117    path!("example.txt"),
118    |file| {
119        file.write(b"Why is black smoke coming out?!")?;
120        file.seek(SeekFrom::End(-24)).unwrap();
121        assert_eq!(file.read(&mut buf)?, 11);
122        Ok(())
123    }
124).unwrap();
125assert_eq!(&buf, b"black smoke");
126```
127*/

181/// A littlefs version number.
182#[derive(Clone, Copy, Debug)]
183pub struct Version(u32);
184
185impl Version {
186    /// The major version component (top-nibble).
187    pub const fn major(&self) -> u16 {
188        let [high, low, _, _] = self.0.to_be_bytes();
189        u16::from_be_bytes([high, low])
190    }
191
192    /// The minor version component (bottom-nibble).
193    pub const fn minor(&self) -> u16 {
194        let [_, _, high, low] = self.0.to_be_bytes();
195        u16::from_be_bytes([high, low])
196    }
197}
198
199impl From<u32> for Version {
200    fn from(version: u32) -> Self {
201        Self(version)
202    }
203}
204
205impl From<Version> for u32 {
206    fn from(version: Version) -> u32 {
207        version.0
208    }
209}
